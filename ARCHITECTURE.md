# 架构说明

## 设计理念

Short Studio 不是一个工具，是一个**编导**。

它有人格、有主动性、有记忆。它会主动给你推选题、帮你盯数据、替你抓评论区。你和它的关系不是"用户调用工具"，而是**创作者和编导搭档**。

飞书是编导的工作平台——多维表格是它的工作台，云文档是稿件存放处，IM 是它和你对话的窗口。

## 整体架构

```
┌─────────────────────────────────────────────────────────┐
│  对话层（飞书 IM 机器人）                                  │
│  编导和创作者在这里聊天、互动、推送通知                      │
├─────────────────────────────────────────────────────────┤
│  Skill 层（本地执行）                                      │
│  /studio → Router → init / topic / script / review /     │
│                      scrape / publish                    │
├──────────────┬──────────────────────────────────────────┤
│  知识层       │  工具层（6 个共享工具）                      │
│  方法论       │  douyin-profile / video2script /          │
│  概念词典     │  douyin-search / trending /               │
│              │  comments / web-search                    │
├──────────────┤         │ HTTP (X-Studio-Key)             │
│  飞书数据层   │  ┌──────▼──────┐                           │
│  多维表格     │  │ Studio API  │  远端服务                  │
│  云文档       │  │ 鉴权+计费    │                           │
│              │  └─────────────┘                           │
└──────────────┴──────────────────────────────────────────┘
```

## 编导角色

编导是 Short Studio 的核心人格。它不是被动的工具，是主动的搭档。

### 编导的日常工作

```
1. 巡逻（主动）
   - 刷热点、刷对标账号 → 发现好选题 → 飞书 IM 推送给创作者
   - 抓评论区 → 汇总粉丝反馈 → 推送分析

2. 陪伴创作（对话驱动）
   - 创作者说"今天写个稿" → 编导拉出待写选题
   - 先出大纲 → 创作者说"第二段换个角度" → 编导调整
   - 全程在飞书 IM 里对话完成

3. 质量把关（主动）
   - 稿子写完 → 编导主动审核
   - "这段 AI 味太重" / "开头 hook 不够强"

4. 数据追踪（主动）
   - 发布后 24h/48h/72h 拉效果数据
   - "这条比上条播放高 3 倍，可能是因为…"
   - 抓评论区 → "粉丝反馈集中在 XX 方面"
```

### 主动触发机制

| 触发方式 | 场景 | 飞书能力 |
|----------|------|----------|
| 定时任务 | 每天早上推荐选题、每周对标采集报告 | IM 消息推送 |
| 事件驱动 | 创作者在 IM 里说话 → 编导响应 | IM 机器人 |
| 状态变更 | 选题状态从"已选"变"写稿中" → 提醒开工 | 多维表格 + IM |
| 数据回流 | 发布后定时拉数据 → 效果播报 | IM 消息推送 |

### 不同部署方式的通知

| 部署方式 | 底层 | 通知通道 |
|----------|------|----------|
| OpenClaw 龙虾 | 龙虾后台 | 飞书 IM 机器人 |
| Claude Code 本地 | 用户终端 | 飞书 IM 机器人 |
| 别人安装 Skill | 别人的 Claude | 飞书 IM 机器人 |

**统一收口到飞书 IM 机器人**——不管底层是什么，通知都发到飞书。创作者只需要盯一个窗口。

## Pipeline

```
初始化（投喂定位 + 关联飞书机器人）
       ↓
┌─────────────┐
│  /studio     │  主入口 Router，编导识别意图
└──────┬──────┘
       ├──→ /studio-init      初始化：定位 + 建表 + 关联机器人
       ├──→ /studio-topic     选题：热点 × 定位匹配
       ├──→ /studio-script    文案：口播稿写作（最重的环节）
       ├──→ /studio-review    审稿：自检 + 去AI味
       ├──→ /studio-scrape    采集：对标账号内容抓取
       └──→ /studio-publish   发布（v2）
```

## 飞书 = 数据库

飞书是编导的数据平台。多维表格存结构化数据，云文档存长内容，IM 做通信。

### 一个 Base，三张表

所有结构化数据在**同一个 Base** 里，这样表之间可以用关联字段互相引用。

#### 表 1：选题库

创作者关心的是：这个选题是什么、从哪来的、现在到哪一步了。

| 字段 | 类型 | 说明 |
|------|------|------|
| 标题 | 文本 | 选题标题 |
| 角度 | 文本 | 切入点，选题阶段就写初步想法 |
| 状态 | 单选 | 待选 / 已选 / 写稿中 / 待审 / 已发布 |
| 来源 | 文本 | 热点链接、参考视频 URL、"自己想的" |
| 备注 | 文本 | 任何补充 |

**表里只存真实数据和状态，不存 AI 判断。** AI 的选题评估在对话里说，创作者看到的表是干净的事实。

#### 表 2：脚本库

| 字段 | 类型 | 说明 |
|------|------|------|
| 标题 | 文本 | 脚本标题 |
| 关联选题 | 关联→选题库 | 这个脚本对应哪个选题 |
| 状态 | 单选 | 大纲 / 初稿 / 终稿 |
| 文档 | 超链接 | 指向飞书云文档（正文太长，不放表里） |
| 审核意见 | 文本 | review 的结论 |

#### 表 3：素材库

这里是**真实数据**——从对标账号采集的内容，创作者能直接看到。

| 字段 | 类型 | 说明 |
|------|------|------|
| 标题 | 文本 | 原视频标题 |
| 作者 | 文本 | 对标账号名 |
| 平台 | 单选 | 抖音 / 小红书 / B站 |
| 链接 | 超链接 | 原始链接 |
| 转写文案 | 文本 | video2script 的结果 |
| 播放量 | 数字 | 平台真实数据 |
| 点赞量 | 数字 | 平台真实数据 |
| 采集时间 | 日期 | |

#### 表 4：发布记录（v2）

| 字段 | 类型 | 说明 |
|------|------|------|
| 关联脚本 | 关联→脚本库 | |
| 平台 | 单选 | 抖音 / 小红书 / B站 |
| 链接 | 超链接 | |
| 播放量 / 点赞 / 评论 | 数字 | 自己的真实效果数据 |

### 飞书云文档

| 文档 | 用途 | 创建时机 |
|------|------|----------|
| 定位文档 | 创作者定位的完整描述 | /studio-init |
| 口播稿 | 每条脚本的正文 | /studio-script |

脚本正文几千字，不适合放表格单元格。放飞书云文档，脚本库里用超链接字段指向它。

### 飞书 IM 机器人

| 能力 | 编导用在哪 |
|------|-----------|
| 主动推送消息 | 推选题、提醒创作、效果播报、粉丝反馈 |
| 接收消息/对话 | 创作者和编导聊天、下指令、给反馈 |
| 通知 | 该创作了、有新热点、数据更新了 |

初始化时关联机器人，之后所有交互都可以在 IM 里发生。创作者不需要记命令，跟编导聊天就行。

### 飞书能力总览

| 飞书能力 | 角色 | lark-cli 命令 |
|----------|------|--------------|
| 多维表格 | 编导的工作台（数据管理） | `lark-cli base +record-*` |
| 云文档 | 稿件存放 | `lark-cli docs +create/+fetch/+update` |
| IM 机器人 | 编导和创作者的对话窗口 | `lark-cli im +messages-send` |

## 共享工具层

多个 Skill 共用的底层能力，统一抽取到 `tools/` 层。

详见 → [tools/TOOLS.md](tools/TOOLS.md)

| 工具 | 能力 | 编导用在哪 |
|------|------|-----------|
| `douyin-profile` | 抖音主页采集 | 采集对标账号 |
| `video2script` | 视频转文案（ASR） | 素材转写 |
| `douyin-search` | 抖音关键词搜索 | 找相关内容 |
| `trending` | 热点趋势 | 选题推荐 |
| `comments` | 抓评论区 | 粉丝反馈、选题灵感、效果判断 |
| `web-search` | 通用搜索（博查等） | 背景调研、事实核查、找论据 |

**关键原则：工具是增强，不是前置依赖。** 工具不可用时，编导降级为手动模式，工厂照样能跑。

## SaaS 服务模型

工具层背后是远端 API 服务，由我们统一运营。

### 用户身份

```
首次使用 → 本地自动生成 UUID → 写入 factory.json
                                     ↓
                              试用额度（100 次）
                                     ↓
                            额度用完 → 弹付款链接
                                     ↓
                          微信支付 → 绑定 UUID ↔ 微信号
                                     ↓
                              换设备 → 微信找回 UUID
```

- **无需注册**：首次使用自动生成身份
- **无需 API Key**：用户不需要自己去申请任何第三方 Key
- **账号恢复**：付费绑定微信后，换设备可通过微信找回

### 服务商模式

```
创作者 → Skill（本地） → Studio API（我们的服务器） → 第三方 API
                          ↑
                    我们持有所有 API Key
                    创作者只需要 Studio Key
```

## 工厂状态管理

### 静态配置（`config/factory.json`）

```json
{
  "user_id": "UUID",
  "creator": { "name": "", "positioning": "", "..." },
  "benchmarks": [],
  "feishu": {
    "positioning_doc": "",
    "base_token": "",
    "tables": {
      "topic": "table_id",
      "script": "table_id",
      "material": "table_id"
    },
    "bot_id": "",
    "chat_id": ""
  }
}
```

初始化时生成，后续可更新。

### 动态数据（飞书多维表格）

- 选题库：选题状态流转
- 脚本库：文案和审核状态
- 素材库：采集的对标内容和真实数据

## Skill 间数据流

```
/studio-init
    → 创建定位云文档
    → 创建 Base（选题库 + 脚本库 + 素材库）
    → 关联飞书机器人
    → 写 factory.json

/studio-scrape
    → 调用 douyin-profile + video2script
    → 写入飞书「素材库」表（真实数据）
    → 本地 JSONL 做缓存/备份

/studio-topic
    → 调用 trending + web-search + douyin-search
    → 读定位文档 + 素材库
    → AI 在对话中推荐 3-5 个选题（评估过程在对话里，不入库）
    → 用户选定 → 写入飞书「选题库」，状态=已选

/studio-script
    → 读「选题库」中状态=已选的选题
    → 读素材库中相关素材（作为参考）
    → 先出大纲 → 用户确认 → 写完整稿
    → lark-cli docs +create 创建飞书文档
    → 飞书「脚本库」新增记录（关联选题 + 文档链接）
    → 选题状态 → 写稿中

/studio-review
    → 读「脚本库」→ 通过文档链接 fetch 正文
    → 去 AI 味 + 方法论校验 + 定位一致性
    → 更新飞书文档 + 写审核意见
    → 脚本状态 → 终稿，选题状态 → 待审

编导主动行为：
    → 定时刷热点 → 飞书 IM 推选题
    → 发布后定时拉数据 → 飞书 IM 播报效果
    → 抓评论区 → 飞书 IM 反馈粉丝意见
```

## 三层知识架构

```
┌─────────────────────────────┐
│  SKILL.md（执行指令层）       │  ← AI 读这层，知道该怎么做
├─────────────────────────────┤
│  方法论（框架层）             │  ← 口播稿框架、去AI味清单、标题公式
├─────────────────────────────┤
│  素材（数据层）               │  ← 飞书素材库（主）+ 本地 JSONL（缓存）
└─────────────────────────────┘
```

- **执行指令层**：每个 skill 的 SKILL.md，定义流程和规则
- **框架层**：`知识库/方法论/` 下的文档，可复用的内容方法论
- **数据层**：飞书素材库（结构化数据，创作者能看到）+ 本地 JSONL（完整转写文案缓存）
