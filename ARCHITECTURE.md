# 架构说明

## 设计理念

Short Studio 是一个**有状态的内容工厂**——它有记忆，有积累，越用越好。

不是"跑一次出结果"的工具，而是一个持续进化的生产系统。

## Pipeline

```
初始化（投喂定位 + 配置）
       ↓
┌─────────────┐
│  /studio     │  主入口 Router，识别意图，路由到子 skill
└──────┬──────┘
       ├──→ /studio-init      初始化：账号定位 + 个人定位 + 平台配置
       ├──→ /studio-topic     选题：热点 × 定位匹配
       ├──→ /studio-script    文案：口播稿写作（最重的环节）
       ├──→ /studio-review    审稿：自检 + 去AI味
       ├──→ /studio-scrape    采集：对标账号内容抓取
       └──→ /studio-publish   发布（v2）
```

## 三层知识架构

借鉴 dbskill 的分层设计，知识分三层：

```
┌─────────────────────────────┐
│  SKILL.md（执行指令层）       │  ← AI 读这层，知道该怎么做
├─────────────────────────────┤
│  方法论（框架层）             │  ← 选题矩阵、口播稿框架、去AI味清单
├─────────────────────────────┤
│  原子库 + 自有库（数据层）    │  ← 对标账号内容、自己的效果数据
└─────────────────────────────┘
```

- **执行指令层**：每个 skill 的 SKILL.md，定义流程和规则
- **框架层**：`知识库/方法论/` 下的文档，可复用的内容方法论
- **数据层**：`知识库/原子库/`（采集）和 `知识库/自有库/`（回流），持续积累

## 工厂状态管理

工厂有两类状态：

### 静态配置（`config/factory.json`）
- 创作者定位（你是谁、做什么方向）
- 平台账号（抖音/小红书/B站地址）
- 对标账号列表
- 飞书多维表格 ID

初始化时生成，后续可更新。

### 动态数据（飞书多维表格）
- 选题库：选题状态流转（待选→已选→写稿中→待审→已发布）
- 脚本库：文案版本管理
- 发布记录：效果数据追踪

## 可升级路线

```
v0.1  骨架 + 初始化 + 定位
v0.2  选题模块
v0.3  文案模块（口播稿）
v0.4  审稿 + 去AI味
v0.5  对标账号采集
---
v1.0  定位→选题→文案 完整闭环
v1.5  效果回流（发布数据反哺选题）
v2.0  声音（TTS / 声音克隆）
v3.0  视频合成（素材剪辑 + 字幕 + 配画面）
v4.0  多平台自动发布 + 数据回收
```

每个版本向后兼容。工厂的"记忆"（知识库 + 多维表格数据）是持续积累的资产，不会因升级丢失。

## 飞书能力使用

| 飞书能力 | 用在哪里 | lark-cli 命令 |
|----------|----------|--------------|
| 多维表格 (base) | 选题库、脚本库、发布记录 | `lark-cli base +record-*` |
| 云文档 (docs) | 定位文档、方法论文档、稿件 | `lark-cli docs +create/+fetch/+update` |
| 即时通讯 (im) | 进度通知、审核提醒 | `lark-cli im +messages-send` |
