# 架构说明

## 设计理念

Short Studio 是一个**有状态的内容工厂**——它有记忆，有积累，越用越好。

不是"跑一次出结果"的工具，而是一个持续进化的生产系统。

## 整体架构

```
┌─────────────────────────────────────────────────────────┐
│  用户层                                                   │
│  /studio → Router → 子 Skill                             │
├─────────────────────────────────────────────────────────┤
│  Skill 层（本地执行）                                      │
│  init / topic / script / review / scrape / publish       │
├──────────────┬──────────────────────────────────────────┤
│  知识层       │  工具层                                    │
│  方法论       │  douyin-profile / video2script /          │
│  原子库       │  douyin-search / trending                 │
│  自有库       │         │                                  │
├──────────────┤         │ HTTP (X-Studio-Key)             │
│  存储层       │  ┌──────▼──────┐                           │
│  factory.json │  │ Studio API  │  远端服务                  │
│  飞书多维表格  │  │ 鉴权+计费    │                           │
│              │  └─────────────┘                           │
└──────────────┴──────────────────────────────────────────┘
```

## Pipeline

```
初始化（投喂定位 + 配置）
       ↓
┌─────────────┐
│  /studio     │  主入口 Router，识别意图，路由到子 skill
└──────┬──────┘
       ├──→ /studio-init      初始化：账号定位 + 个人定位 + 平台配置
       ├──→ /studio-topic     选题：热点 × 定位匹配
       ├──→ /studio-script    文案：口播稿写作（最重的环节）
       ├──→ /studio-review    审稿：自检 + 去AI味
       ├──→ /studio-scrape    采集：对标账号内容抓取
       └──→ /studio-publish   发布（v2）
```

## 三层知识架构

借鉴 dbskill 的分层设计，知识分三层：

```
┌─────────────────────────────┐
│  SKILL.md（执行指令层）       │  ← AI 读这层，知道该怎么做
├─────────────────────────────┤
│  方法论（框架层）             │  ← 选题矩阵、口播稿框架、去AI味清单
├─────────────────────────────┤
│  原子库 + 自有库（数据层）    │  ← 对标账号内容、自己的效果数据
└─────────────────────────────┘
```

- **执行指令层**：每个 skill 的 SKILL.md，定义流程和规则
- **框架层**：`知识库/方法论/` 下的文档，可复用的内容方法论
- **数据层**：`知识库/原子库/`（采集）和 `知识库/自有库/`（回流），持续积累

## 共享工具层

多个 Skill 共用的底层能力，统一抽取到 `tools/` 层。

详见 → [tools/TOOLS.md](tools/TOOLS.md)

| 工具 | 能力 | 调用方 |
|------|------|--------|
| `douyin-profile` | 抖音主页采集 | scrape, init |
| `video2script` | 视频转文案（ASR） | scrape, script |
| `douyin-search` | 关键词搜索 | topic, scrape |
| `trending` | 热点趋势 | topic |

**关键原则：工具是增强，不是前置依赖。** 工具不可用时，Skill 降级为手动输入模式，工厂照样能跑。

## SaaS 服务模型

工具层背后是远端 API 服务（`skill-api/`），由我们统一运营：

### 用户身份

```
首次使用 → 本地自动生成 UUID → 写入 factory.json
                                     ↓
                              试用额度（100 次）
                                     ↓
                            额度用完 → 弹付款链接
                                     ↓
                          微信支付 → 绑定 UUID ↔ 微信号
                                     ↓
                              换设备 → 微信找回 UUID
```

- **无需注册**：首次使用自动生成身份
- **无需 API Key**：用户不需要自己去申请任何第三方 Key
- **账号恢复**：付费绑定微信后，换设备可通过微信找回

### 计费模型

- 统一额度池，所有工具共享
- 每个工具调用消耗固定 credits（1-2 credits/次）
- 新用户免费试用额度
- 充值包：按额度购买

### 服务商模式

```
用户 → Skill（本地） → Studio API（我们的服务器） → 第三方 API（抖音/ASR/...）
                          ↑
                    我们持有所有 API Key
                    用户只需要 Studio Key
```

好处：
- 用户零配置，开箱即用
- 我们控制成本和质量
- 可以缓存热门数据，降低调用成本
- 一个入口管理所有第三方依赖

## 工厂状态管理

工厂有两类状态：

### 静态配置（`config/factory.json`）
- `user_id`：UUID，工具层鉴权用
- 创作者定位（你是谁、做什么方向）
- 平台账号（抖音/小红书/B站地址）
- 对标账号列表
- 飞书多维表格 ID

初始化时生成，后续可更新。

### 动态数据（飞书多维表格）
- 选题库：选题状态流转（待选→已选→写稿中→待审→已发布）
- 脚本库：文案版本管理
- 发布记录：效果数据追踪

## Skill 间数据流

```
/studio-init
    ↓ 写入 factory.json + 飞书定位文档
    ↓
/studio-topic
    ↓ 读 factory.json（定位）+ 调用 trending/douyin-search
    ↓ 写入飞书多维表格「选题库」，状态=待选
    ↓
/studio-script  ← 读「选题库」中状态=已选的选题
    ↓ 读 知识库/方法论/ + 知识库/原子库/（参考文案）
    ↓ 先出大纲 → 用户确认 → 写完整稿
    ↓ 写入飞书云文档 + 更新选题状态=写稿中→待审
    ↓
/studio-review  ← 读飞书云文档中的稿件
    ↓ 对照 去AI味清单 + 口播稿框架 + 定位
    ↓ 输出审核报告 + 修改稿 → 更新飞书文档
    ↓ 更新选题状态=已审
    ↓
/studio-scrape（独立运行，喂入原子库）
    ↓ 调用 douyin-profile + video2script
    ↓ 写入 知识库/原子库/*.jsonl
```

选题状态流转：`待选 → 已选 → 写稿中 → 待审 → 已审 → 已发布`

## 飞书能力使用

| 飞书能力 | 用在哪里 | lark-cli 命令 |
|----------|----------|--------------|
| 多维表格 (base) | 选题库、脚本库、发布记录 | `lark-cli base +record-*` |
| 云文档 (docs) | 定位文档、方法论文档、稿件 | `lark-cli docs +create/+fetch/+update` |
| 即时通讯 (im) | 进度通知、审核提醒 | `lark-cli im +messages-send` |

## 已识别的架构问题与解决方案

### 1. factory.json 与云端同步

**问题：** factory.json 是本地文件，用户换设备或多设备使用时数据丢失。

**方案：** v1.0 阶段 factory.json 纯本地。v2.0 考虑将核心配置（定位、对标列表）同步到 Studio API，UUID 作为同步 key。飞书多维表格天然是云端的，不受影响。

### 2. 工具调用失败处理

**问题：** 网络问题、API 限流、额度不足。

**方案：**
- 每个工具调用前先检查 `factory.json` 中有无 `user_id`，没有就生成
- 调用失败时，Skill 输出明确错误信息 + 降级方案
- 额度不足时返回付款链接，不要死循环重试

### 3. 原子库数据格式一致性

**问题：** 不同工具返回的数据格式不一致，原子库难以统一检索。

**方案：** 原子库统一用 JSONL 格式，每条记录必须包含：
```json
{"source": "douyin", "type": "video", "author": "...", "title": "...", "content": "...", "url": "...", "scraped_at": "..."}
```

### 4. 知识库膨胀

**问题：** 原子库持续采集，数据量会越来越大。

**方案：**
- 原子库按对标账号分文件（`原子库/{account_name}.jsonl`）
- 定期归档：超过 6 个月的数据移入 `原子库/archive/`
- Skill 检索时只读近 3 个月数据，除非用户指定

### 5. 多用户隔离

**问题：** Skill 包是开源的，多用户数据不能混。

**方案：** 每个用户在自己的机器上运行，数据天然隔离。factory.json、知识库、飞书多维表格都是 per-user 的。Studio API 层通过 UUID 隔离计费。

## 可升级路线

```
v0.1  骨架 + 初始化 + 定位
v0.2  选题模块
v0.3  文案模块（口播稿）
v0.4  审稿 + 去AI味
v0.5  对标账号采集
---
v1.0  定位→选题→文案 完整闭环
v1.5  效果回流（发布数据反哺选题）
v2.0  声音（TTS / 声音克隆）+ 配置云端同步
v3.0  视频合成（素材剪辑 + 字幕 + 配画面）
v4.0  多平台自动发布 + 数据回收
```

每个版本向后兼容。工厂的"记忆"（知识库 + 多维表格数据）是持续积累的资产，不会因升级丢失。
